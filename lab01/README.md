# Лабораторная работа №2

## Паттерн «Компоновщик» (Composite): Система управления складом

### Предметная область

Паттерн **«Компоновщик» (Composite)** широко применяется в разработке приложений, где требуется работать с иерархическими структурами, где отдельные объекты и их группы должны обрабатываться единообразно. Один из примеров — система управления складской логистикой.

**Система управления складом** — это компонент приложения, который отвечает за хранение информации о грузах, контейнерах и их иерархической структуре, а также за расчёт суммарных характеристик (вес, стоимость страховки) для любого уровня вложенности.

### Описание проблемы

В складской логистике существуют различные типы объектов:

- **Груз** — неделимый объект, имеет вес, стоимость страховки, штрих-код;
- **Контейнер** — может содержать грузы и другие контейнеры, имеет собственный идентификатор и тип;
- **Склад** — корневой элемент, содержит контейнеры и отдельные грузы.

Если клиентский код (например, GUI приложения) будет напрямую работать с каждым типом объектов, возникнут проблемы:

1. **Разнородные списки** — склад вынужден хранить два разных списка: для контейнеров и для отдельных грузов. Контейнер также хранит два списка: для грузов и для вложенных контейнеров;
2. **Дублирование кода** — для каждой операции (расчёт веса, поиск) приходится писать отдельную логику для каждого типа;
3. **Сложность расширения** — при добавлении нового типа (например, «Паллета» или «Сейф-пакет») придётся изменять существующий код;
4. **Нарушение принципа Open/Closed** — классы не закрыты для модификации;
5. **Сложность рекурсивных операций** — поиск по штрих-коду или расчёт общего веса требуют сложной рекурсивной логики с разветвлениями по типам.

### Решение проблемы

Использование **паттерна «Компоновщик» (Composite)** для представления иерархической структуры склада.

Паттерн гарантирует, что:

1. **Единый интерфейс** — все элементы (груз, контейнер, склад) реализуют общий интерфейс `CargoComponent`;
2. **Однородные списки** — контейнер и склад хранят один список `vector<shared_ptr<CargoComponent>>`, что упрощает все операции;
3. **Рекурсивные операции** — методы `getWeight()`, `getInsurance()`, `findByBarcode()` автоматически рекурсивно обходят всю иерархию;
4. **Добавление нового типа** не требует изменения существующего кода — достаточно создать новый класс, реализующий интерфейс `CargoComponent`;
5. Все компоненты следуют единому интерфейсу, что обеспечивает полиморфную работу.

## Ключевые особенности реализации

### Динамическое добавление грузов и контейнеров

В версии с паттерном пользователь может добавлять **любые** грузы и контейнеры через графический интерфейс:

1. **Добавление груза:**
   - Нажать кнопку **"ДОБАВИТЬ ГРУЗ"**
   - Ввести:
     - Название груза (например, "Ноутбуки Dell", "Мониторы Samsung")
     - Вес (кг)
     - Страховку (руб/кг)
     - Штрих-код
     - Категорию (Обычный, Хрупкий, Опасный, Скоропортящийся, Оборудование)
   - Выбрать размещение (склад или конкретный контейнер)

2. **Добавление контейнера:**
   - Нажать кнопку **"ДОБАВИТЬ КОНТЕЙНЕР"**
   - Ввести:
     - Название контейнера (например, "Морской контейнер 20ft")
     - Тип контейнера (Стандартный, Рефрижератор, Опасный, Составной)

3. **Поиск по штрих-коду:**
   - Ввести штрих-код (например, "BC-001")
   - Нажать кнопку **"ПОИСК"**
   - Система рекурсивно находит груз по всей иерархии склада

### Преимущества реализации

1. **Расширяемость** — новые типы грузов и контейнеров добавляются без изменения кода
2. **Инкапсуляция** — логика рекурсивных операций скрыта внутри классов
3. **Полиморфизм** — все компоненты работают через единый интерфейс `CargoComponent`
4. **Гибкость** — пользователь сам определяет иерархию вложения контейнеров
5. **Единообразие** — клиентский код не различает листья и композиты

## Вывод

Паттерн «Компоновщик» позволил создать гибкую, расширяемую систему управления складом, в которой все элементы иерархии обрабатываются единообразно. Это наглядно демонстрирует преимущества паттерна:

- **Слабая связанность** компонентов
- **Возможность динамического расширения** иерархии
- **Инкапсуляция** рекурсивных операций
- **Возможность добавления новых типов** без изменения существующего кода
- **Единообразная обработка** отдельных объектов и их групп

Версия без паттерна, напротив, показывает все проблемы жесткой связанности, разнородных списков и невозможности расширения без изменения существующего кода.
