# Лабораторная работа №3

## Паттерн «Цепочка обязанностей» (Chain of Responsibility): RPG игра «Dragon Quest»

### Предметная область

Паттерн **«Цепочка обязанностей» (Chain of Responsibility)** широко применяется в разработке игр, где требуется последовательная проверка множества условий перед выполнением действия. Один из примеров — система проверки выполнения квеста в RPG-игре.

**Система проверки квестов** — это компонент игры, который отвечает за последовательную проверку всех условий для завершения квеста: уровень персонажа, наличие предметов, выполненные предыдущие квесты, репутация и другие параметры.

### Описание проблемы

В RPG-игре для завершения квеста игрок должен выполнить несколько условий:

- **Достаточный уровень** — персонаж должен быть определённого уровня;
- **Наличие предмета** — в инвентаре должен быть определённый предмет (например, «Меч героя»);
- **Выполнение предыдущих квестов** — игрок должен пройти предшествующие квесты;
- **Достаточная репутация** — репутация с определённой фракцией должна быть не ниже требуемого значения.

Если клиентский код (например, игровой движок) будет напрямую проверять все условия с помощью вложенных `if-else`, возникнут проблемы:

1. **Жёсткая связанность** — код игры зависит от всех конкретных условий;
2. **Сложность изменения порядка проверок** — при изменении порядка придётся переписывать логику;
3. **Дублирование кода** — проверки разбросаны по разным местам;
4. **Нарушение принципа Open/Closed** — при добавлении нового условия (например, проверка времени суток или случайной удачи) придётся изменять существующий код.

### Решение проблемы

Использование **паттерна «Цепочка обязанностей» (Chain of Responsibility)** для проверки условий выполнения квеста.

Паттерн гарантирует, что:

1. Каждый обработчик отвечает за проверку одного конкретного условия;
2. Если условие выполнено — обработчик передаёт управление следующему в цепочке;
3. Если условие НЕ выполнено — цепочка прерывается, и квест считается невыполненным;
4. Клиентский код не знает, какие именно проверки будут выполняться;
5. Цепочку можно легко изменять: добавлять новые обработчики, менять порядок, удалять существующие.

## Ключевые особенности реализации

### Построение цепочки обработчиков

В версии с паттерном каждый квест строит свою цепочку обработчиков:

1. **LevelHandler** — проверяет уровень персонажа
2. **ItemHandler** — проверяет наличие предмета в инвентаре
3. **PreviousQuestHandler** — проверяет выполнение предыдущих квестов
4. **ReputationHandler** — проверяет репутацию с фракцией
5. **RewardHandler** — выдаёт награду при успешном прохождении всех проверок

Соединение в цепочку:
level_check.set_next(item_check).set_next(quest_check).set_next(rep_check).set_next(reward_check)

### Игровой мир

В версии с паттерном реализован полноценный игровой мир:

1. **Перемещение по карте** — игрок может ходить по деревне с помощью стрелок или кнопок;
2. **Взаимодействие с NPC** — диалоги с жителями, выполнение квестов;
3. **Боевая система** — атака врагов (волков и браконьеров) с ответной атакой;
4. **Сбор ресурсов** — добыча железной руды для создания меча;
5. **Битва с боссом** — финальное сражение с драконом в отдельном окне с анимацией;
6. **Система репутации** — повышение репутации через выполнение заданий.

### Преимущества реализации

1. **Расширяемость** — новые условия добавляются без изменения существующего кода;
2. **Инкапсуляция** — логика каждой проверки скрыта внутри своего обработчика;
3. **Гибкость** — порядок проверок можно легко изменить;
4. **Переиспользование** — одни и те же обработчики можно использовать для разных квестов;
5. **Наглядность** — GUI показывает статус каждого условия в реальном времени.

### Вывод

Паттерн «Цепочка обязанностей» позволил создать гибкую, расширяемую систему проверки выполнения квестов, в которой каждое условие инкапсулировано в отдельном классе-обработчике. Это наглядно демонстрирует преимущества паттерна:

- **Слабая связанность** между проверками;
- **Возможность динамического изменения** порядка и состава проверок;
- **Лёгкость добавления** новых типов условий;
- **Упрощение клиентского кода** — он не знает о внутренней структуре проверок.

Версия без паттерна, напротив, показывает все проблемы жесткой связанности, когда все проверки зашиты в одном методе с множеством вложенных `if-else`, что делает код сложным для поддержки и расширения.
