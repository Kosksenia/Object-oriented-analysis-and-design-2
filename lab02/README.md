# Лабораторная работа №2

## Паттерн «Компоновщик» (Composite): Система управления складом

### Предметная область

Паттерн **«Компоновщик» (Composite)** широко применяется в разработке приложений, где требуется работать с иерархическими структурами, в которых отдельные объекты и их группы должны обрабатываться единообразно. Классический пример — система управления складской логистикой.

**Система управления складом** — это компонент приложения, который отвечает за хранение информации о грузах, контейнерах и их иерархической структуре, а также за расчёт суммарных характеристик (вес, стоимость страховки) для любого уровня вложенности.

### Описание проблемы

В складской логистике существуют различные типы объектов:

- **Груз** — неделимый объект, имеет вес, стоимость страховки, штрих-код;
- **Контейнер** — может содержать грузы и другие контейнеры, имеет собственный идентификатор и тип;
- **Склад** — корневой элемент, содержит контейнеры и отдельные грузы.

Если клиентский код (например, GUI приложения) будет напрямую работать с каждым типом объектов, возникают следующие **проблемы**:

1. **Разнородные списки** — склад вынужден хранить два разных списка: для контейнеров и для отдельных грузов. Контейнер также хранит два списка: для грузов и для вложенных контейнеров.
2. **Дублирование кода** — для каждой операции (расчёт веса, поиск) приходится писать отдельную логику для каждого типа.
3. **Сложность расширения** — при добавлении нового типа (например, «Паллета» или «Сейф-пакет») потребуется изменять все существующие классы.
4. **Нарушение принципа Open/Closed** — классы не закрыты для модификации.
5. **Сложность рекурсивных операций** — поиск по штрих-коду или расчёт общего веса требуют сложной рекурсивной логики с разветвлениями по типам.

### Решение проблемы

Использование **паттерна «Компоновщик» (Composite)** для представления иерархической структуры склада.

Паттерн гарантирует, что:

1. **Единый интерфейс** — все элементы (груз, контейнер, склад) реализуют общий интерфейс `CargoComponent`.
2. **Однородные списки** — контейнер и склад хранят один список `std::vector<std::shared_ptr<CargoComponent>>`, что упрощает все операции.
3. **Рекурсивные операции** — методы `getWeight()`, `getInsurance()`, `findByBarcode()` автоматически рекурсивно обходят всю иерархию.
4. **Простота расширения** — добавление нового типа (например, «Паллета») требует только создания нового класса, реализующего интерфейс `CargoComponent`.
5. **Прозрачность для клиента** — клиентский код работает только с интерфейсом `CargoComponent`, не различая листья и композиты.

На диаграмме показана структура паттерна:
- **`CargoComponent`** — абстрактный базовый класс (интерфейс), определяющий общие методы для всех элементов иерархии.
- **`Cargo`** — листовой элемент (Leaf), представляет отдельный груз.
- **`Container`** — композит (Composite), может содержать другие компоненты.
- **`Warehouse`** — композит (Composite), корневой элемент иерархии, представляет склад.
- **`GUI`** — клиентский код, который работает с иерархией через единый интерфейс.

### Ключевые особенности реализации

#### Единый интерфейс

Все классы (`Cargo`, `Container`, `Warehouse`) наследуются от `CargoComponent`, что позволяет обрабатывать их единообразно:

```cpp
// Единый интерфейс для всех элементов
class CargoComponent {
public:
    virtual double getWeight() const = 0;
    virtual double getInsurance() const = 0;
    virtual void add(std::shared_ptr<CargoComponent> comp) {}
    // ...
};
